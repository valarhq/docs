The `valar` package bundles two product surfaces behind one import. Whether you
are recording what an agent did or calling a hosted model, you start the same
way:

```python theme={null}
import valar
```

From there, `valar.trace` and `valar.inference` give you the two surfaces, and
`valar.Config` plus the Tinker helpers support them. Read on for setup, then
jump to whichever surface you need.

## What each surface does

## Install the package

Pick whichever installer your project uses:

```bash theme={null}
pip install valar-sdk
```

```bash theme={null}
uv add valar-sdk
```

The SDK targets Python 3.11 and later.

## Set your credentials

The SDK reads its API key from the environment. Export it once per shell:

```bash theme={null}
export VALAR_API_KEY=sk_live_8f2c00a1b9
```

How a missing key is handled depends on the surface:

* **Inference** needs `VALAR_API_KEY`. Without it, the call fails.
* **Traces** degrade gracefully. When the key is absent, `valar.trace` becomes
  a no-op — nothing is recorded and no network request is made — so local
  scripts keep running. See the [Traces reference](/traces-sdk) for the
  details.

To point the SDK at a non-production environment, set `VALAR_MODE`. It accepts
`prod` (the default), `dev`, and `staging`. The complete list of variables the
SDK reads is in the
[Traces environment table](/traces-sdk#environment-variables).

## Public symbols

These are exported at the top level of the `valar` module:

| Symbol | Purpose | Reference |
| ------ | ------- | --------- |
| `valar.trace` | Record agent and task trajectories | [Traces](/traces-sdk) |
| `valar.inference` | Trace-correlated inference wrappers | [Inference](/traces-sdk-inference) |
| `valar.Config` | Resolved endpoint settings | [Configuration](/sdk-configuration) |
| `valar.RetryPolicy` | Retry and backoff behavior | [Configuration](/sdk-configuration) |
| `valar.ValarTokenCompleter` | Tinker token sampler | [Tinker](/sdk-tinker) |
| `valar.get_tinker_checkpoint_signed_url` | Resolve a Tinker checkpoint URL | [Tinker](/sdk-tinker) |

Read the installed version from `valar.__version__`.

## How errors are organized

`valar.ValarError` sits at the root of the SDK's exception tree, so wrapping a
call in `except valar.ValarError` catches anything the SDK raises. Each surface
narrows the hierarchy from there — see
[Traces → Errors](/traces-sdk-errors) for trace and inference exceptions.

Some methods deliberately raise built-in Python exceptions instead —
`ValueError`, `LookupError`, `PermissionError`, `FileNotFoundError`, or
`TimeoutError` — when one of those fits the situation better. Those methods note
it where it applies.
