---
title: "Completion Windows"
description: "Trade turn latency for cost by choosing the completion window that matches your trajectory's time budget"
---

Valar optimizes for throughput and cost per token, not the latency of one turn. In agentic workloads the number that matters is how long a full trajectory takes end to end, not the round-trip of a single call. A completion window tells Valar how much wall-clock time per turn your workload can tolerate, and you pay less the more time you can give it.

## Set the window

Pass `metadata.completion_window` on the request:

```py
response = client.responses.create(
    model="zai-org/GLM-5.1-FP8",
    input="Explain the key ideas behind transformers.",
    background=True,
    metadata={
        "completion_window": "priority"
    }
)
```

Accepted values are `"asap"` (the **Now** tier), `"priority"`, `"standard"`, and `"flex"`.

## The four tiers

| Window | Avg. turn time | Price vs Now | Best for |
| --- | --- | --- | --- |
| Now | Immediate | — | Interactive UIs, human-in-the-loop |
| `priority` | ~1 min | ~50% lower | Latency-sensitive agent loops |
| `standard` | ~5 min | ~65% lower | Cost-optimized agents |
| `flex` | best-effort | ~80% lower | Batch processing, evals, offline |

<Note>
  Actual turn times vary with the specific workload and the model you choose.
</Note>

Each model-and-window price pairing is listed on the [Pricing](/pricing) page.

## How each tier behaves

**Now** runs immediately on the fastest hardware in a latency-optimized setup. Valar does not focus its optimization here; the tier exists as a convenience and is priced in line with other model API providers.

`priority` is the most tightly scheduled tier. For a higher per-token rate it targets a lower average turn time than `standard` — roughly 1 minute against `standard`'s ~5 on a balanced workload. Use it for latency-sensitive agent loops where one turn feeds the next and several extra minutes per turn would noticeably stretch the whole trajectory.

`standard` is the default wherever a model supports it. It runs on Valar's maximum-efficiency serving stack and targets roughly a five-minute average turn time across a balanced workload. Most of Valar's published prices reference this window.

`flex` runs your work whenever compute is cheapest — overnight or off-peak — with no response-time target. It fits batch and non-agentic jobs.

<Warning>
  `flex` requires `background=True`.
</Warning>

## What happens by default

Leave `completion_window` off and the request defaults to `standard`. When `standard` is not available for the chosen model, <Tooltip tip="Responses API with `background=True`, or the Batch API">async</Tooltip> requests fall back to `flex` where it exists, and <Tooltip tip="Responses without `background=True`, Chat Completions, Messages, or async requests using models without `flex` available">every other request</Tooltip> falls back to the **Now** tier.

<Warning>
  Explicitly choosing a window the model does not support fails with `400 invalid_request_error`. The error message lists the model's supported windows, and the [Pricing](/pricing) page keeps a current support matrix.
</Warning>
