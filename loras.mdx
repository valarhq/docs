ValarAI lets you bring PEFT-trained LoRAs and run them against supported base models. Once an adapter is uploaded, assign it a name and reference that name (or its ID) through request metadata.

## Supported base models

<Note>
  LoRA serving is presently in a closed pilot limited to select partners. If you
  want to fine-tune and serve LoRAs on ValarAI, [reach
  out](mailto:support@valarhq.ai).
</Note>

The following base models can serve LoRAs:

| Base model             | Max rank | Target modules |
| ---------------------- | -------- | -------------- |
| `moonshotai/Kimi-K2.6` | 32       | all            |

When an adapter doesn't match the base model, or its rank goes above the limit, either validation or inference will fail.

## Adapter requirements

Fine-tune using [PEFT](https://huggingface.co/docs/peft) and ship the two standard artifacts:

* `adapter_config.json`
* `adapter_model.safetensors`

Your adapter config needs values that line up with the base model:

* **`base_model_name_or_path`** should point at the base model registered in `supported_models` (for example, `moonshotai/Kimi-K2.6`).
* **`peft_type`** must be `"LORA"`.
* **`task_type`** must be `"CAUSAL_LM"`.
* **`r`** (the rank) cannot exceed the base model's max rank.
* **`target_modules`** may name any modules that the base model and runtime support. For Kimi K2.6 LoRAs, ValarAI imposes no fixed target-module allowlist.

The remaining adapter-config fields — `lora_alpha`, `lora_dropout`, bias settings, and so on — carry through unchanged.

Each of `adapter_config.json` and `adapter_model.safetensors` may be as large as 100 GiB.

## Add a LoRA

Adding a LoRA takes three steps: upload the config, upload the weights, and then issue a `POST /v1/loras`. The response carries a validation record per requested model.

### 1. Upload the two adapter files

Send both with `POST /v1/files` (multipart):

```python theme={null}
from openai import OpenAI

client = OpenAI(base_url="https://api.valarhq.ai/v1", api_key="YOUR_KEY")

with open("my-adapter/adapter_config.json", "rb") as f:
    cfg = client.files.create(file=f, purpose="lora")

with open("my-adapter/adapter_model.safetensors", "rb") as f:
    wts = client.files.create(file=f, purpose="lora")
```

### 2. Create the LoRA

```python theme={null}
import requests

resp = requests.post(
    "https://api.valarhq.ai/v1/loras",
    headers={"Authorization": "Bearer YOUR_KEY"},
    json={
        "name": "funnier-v1",
        "supported_models": ["moonshotai/Kimi-K2.6"],
        "config_file_id": cfg.id,
        "weights_file_id": wts.id,
        # optional
        "display_name": "Funnier v1",
        "description": "Fine-tuned on a standup-comedy corpus to write punchier, joke-forward replies.",
    },
    timeout=30,
)
resp.raise_for_status()
lora = resp.json()
print(lora["id"], lora["status"])  # e.g. 3fa85f64-... pending_validation
```

Every value in `supported_models` has to be a recognized ValarAI model ID.

The `name` field follows these rules:

* between 2 and 64 characters
* lowercase alphanumerics or dashes (`[a-z0-9-]`)
* the first and last characters must be alphanumeric
* unique within your organization (a duplicate returns `409`)

The file IDs you supply have to belong to the same organization as your API key.

### Validation flow

For each entry in `supported_models`, `POST /v1/loras` produces one validation record. These records show up in the response under `validations`:

```json theme={null}
{
  "id": "3fa85f64-...",
  "name": "funnier-v1",
  "status": "pending_validation",
  "supported_models": ["moonshotai/Kimi-K2.6"],
  "validations": [
    {
      "model": "moonshotai/Kimi-K2.6",
      "status": "pending",
      "response_id": "resp_..."
    }
  ]
}
```

ValarAI runs validation against every model listed in `supported_models`. Poll `GET /v1/loras/{name}` until the model you intend to use has finished validating.

| Validation status | Meaning                                                                                            |
| ----------------- | -------------------------------------------------------------------------------------------------- |
| `pending`         | The validation task has been created.                                                              |
| `running`         | The validation request is underway.                                                                |
| `succeeded`       | The LoRA loaded and finished a validation request for that model.                                  |
| `failed`          | The LoRA can't be used for that model. See `result_code` and `result_message` for the failure.     |
| `unverified`      | ValarAI was unable to validate automatically.                                                      |

Check `validations` for per-model status. A request that uses a LoRA is only rejected when that model's most recent validation is `failed`; records in `pending`, `running`, or `unverified` states stay usable.

Should you later attach a model via `PATCH /v1/loras/{name}`, ValarAI generates validation records for the newly added models that haven't already passed validation.

### 3. Fetch LoRAs

```bash theme={null}
# by name
curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras/funnier-v1

# by id
curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras/3fa85f64-...

# list all loras for your org
curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras
```

## Use a LoRA

On any Responses, Chat Completions, or Messages request, set the LoRA's name (or its UUID) as `metadata.lora`. The `model` you choose has to appear in that LoRA's `supported_models`:

```python theme={null}
response = client.responses.create(
    model="moonshotai/Kimi-K2.6",
    input=[{"role": "user", "content": "Write a one-liner about a centrifuge that's having a bad day."}],
    metadata={
        "lora": "funnier-v1",
        "completion_window": "priority",
    },
    background=True,
)
```

Chat Completions:

```python theme={null}
response = client.chat.completions.create(
    model="moonshotai/Kimi-K2.6",
    messages=[{"role": "user", "content": "Tell me a joke about Go channels."}],
    extra_body={"metadata": {"lora": "funnier-v1", "completion_window": "priority"}},
)
```

### Constraints on LoRA requests

* **`completion_window` can't be `asap`.** Pick `priority`, `standard`, or `flex` — the [Completion Windows](/completion-windows) page explains each one.
* **`model` has to be listed in the LoRA's `supported_models`.** Asking for any other base model yields a `400`.
* **A failed model validation blocks that model.** When validation fails, `GET /v1/loras/{name}` surfaces `validations[].result_message`.
* **The LoRA has to be owned by your organization.** Names are per-org, so two organizations can each own a separate LoRA named `funnier-v1`.
* **In `metadata.lora`, either the LoRA's name or its UUID works.**
