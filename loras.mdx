---
title: "LoRAs"
description: "Serve your own PEFT-trained LoRA adapters against compatible Valar base models"
---

If you have trained a LoRA with PEFT, you can host it on Valar and route requests through it. The flow is: upload the adapter, give it a name, let Valar validate it against each base model you want, then reference that name (or its ID) in request metadata at inference time.

<Note>
  LoRA serving is currently a closed pilot open to select partners. To fine-tune and serve LoRAs on Valar, [reach out](mailto:support@valarhq.ai).
</Note>

## What you can serve it on

These base models accept LoRAs today. The rank ceiling and the modules you may target are fixed per base model:

| Base model | Max rank | Target modules |
| ---------- | -------- | -------------- |
| `moonshotai/Kimi-K2.6` | 32 | all |

If an adapter does not match its base model, or its rank climbs above the ceiling, it fails at either validation or inference.

## What the adapter must contain

Train with [PEFT](https://huggingface.co/docs/peft) and produce the two standard artifacts:

- `adapter_config.json`
- `adapter_model.safetensors`

<Note>
  Each of the two files may be up to 100 GiB.
</Note>

Inside `adapter_config.json`, the following fields have to agree with the base model:

<ParamField path="base_model_name_or_path" type="string">
  Must point at a base model registered in `supported_models`, for example `moonshotai/Kimi-K2.6`.
</ParamField>

<ParamField path="peft_type" type="string">
  Must be `"LORA"`.
</ParamField>

<ParamField path="task_type" type="string">
  Must be `"CAUSAL_LM"`.
</ParamField>

<ParamField path="r" type="integer">
  The rank. Must not exceed the base model's max rank.
</ParamField>

<ParamField path="target_modules" type="array">
  Any modules the base model and runtime support. Kimi K2.6 LoRAs have no fixed target-module allowlist.
</ParamField>

Everything else in the config — `lora_alpha`, `lora_dropout`, bias settings, and the rest — passes through untouched.

## Add a LoRA

<Steps>
  <Step title="Upload both adapter files">
    Send each file to `POST /v1/files` (multipart) with `purpose="lora"`:

    ```python theme={null}
    from openai import OpenAI

    client = OpenAI(base_url="https://api.valarhq.ai/v1", api_key="YOUR_KEY")

    with open("my-adapter/adapter_config.json", "rb") as f:
        cfg = client.files.create(file=f, purpose="lora")

    with open("my-adapter/adapter_model.safetensors", "rb") as f:
        wts = client.files.create(file=f, purpose="lora")
    ```

    Both file IDs must belong to the same organization as your API key.
  </Step>

  <Step title="Register the LoRA">
    Call `POST /v1/loras` with the two file IDs and the models you want to run on:

    ```python theme={null}
    import requests

    resp = requests.post(
        "https://api.valarhq.ai/v1/loras",
        headers={"Authorization": "Bearer YOUR_KEY"},
        json={
            "name": "punchier-v1",
            "supported_models": ["moonshotai/Kimi-K2.6"],
            "config_file_id": cfg.id,
            "weights_file_id": wts.id,
            # optional
            "display_name": "Punchier v1",
            "description": "Tuned on a standup-comedy corpus for joke-forward replies.",
        },
        timeout=30,
    )
    resp.raise_for_status()
    lora = resp.json()
    print(lora["id"], lora["status"])  # e.g. 3fa85f64-... pending_validation
    ```

    Constraints on the request:

    - Every entry in `supported_models` must be a recognized Valar model ID.
    - `name` is 2 to 64 characters, lowercase `[a-z0-9-]`, must start and end with an alphanumeric, and must be unique within your organization — a duplicate returns `409`.
    - The file IDs must belong to your organization.
  </Step>

  <Step title="Wait for validation to finish">
    `POST /v1/loras` creates one validation record per entry in `supported_models`, returned under `validations`:

    ```json theme={null}
    {
      "id": "3fa85f64-...",
      "name": "punchier-v1",
      "status": "pending_validation",
      "supported_models": ["moonshotai/Kimi-K2.6"],
      "validations": [
        {
          "model": "moonshotai/Kimi-K2.6",
          "status": "pending",
          "response_id": "resp_..."
        }
      ]
    }
    ```

    Valar validates against every listed model. Poll `GET /v1/loras/{name}` until the model you plan to use has finished:

    | Validation status | Meaning |
    | ----------------- | ------- |
    | `pending` | The validation task has been created. |
    | `running` | The validation request is underway. |
    | `succeeded` | The LoRA loaded and completed a validation request for that model. |
    | `failed` | The LoRA cannot be used for that model. Check `result_code` and `result_message`. |
    | `unverified` | Valar could not validate automatically. |

    A request using the LoRA is rejected only when that model's most recent validation is `failed`. Records in `pending`, `running`, or `unverified` remain usable.

    <Note>
      Attach more models later with `PATCH /v1/loras/{name}`. Valar creates validation records for any newly added models that have not already passed.
    </Note>
  </Step>

  <Step title="Read back your LoRAs">
    ```bash theme={null}
    # by name
    curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras/punchier-v1

    # by id
    curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras/3fa85f64-...

    # all loras for your org
    curl -H "Authorization: Bearer $VALAR_API_KEY" https://api.valarhq.ai/v1/loras
    ```
  </Step>
</Steps>

## Run a request through a LoRA

Set the LoRA's name or its UUID as `metadata.lora` on any Responses, Chat Completions, or Messages request. The `model` you pick must appear in that LoRA's `supported_models`.

<Tabs>
  <Tab title="Responses">
    ```python theme={null}
    response = client.responses.create(
        model="moonshotai/Kimi-K2.6",
        input=[{"role": "user", "content": "Write a one-liner about a centrifuge having a bad day."}],
        metadata={
            "lora": "punchier-v1",
            "completion_window": "standard",
        },
        background=True,
    )
    ```
  </Tab>

  <Tab title="Chat Completions">
    ```python theme={null}
    response = client.chat.completions.create(
        model="moonshotai/Kimi-K2.6",
        messages=[{"role": "user", "content": "Tell me a joke about Go channels."}],
        extra_body={"metadata": {"lora": "punchier-v1", "completion_window": "standard"}},
    )
    ```
  </Tab>
</Tabs>

<Warning>
  A LoRA request can't use the **Now** (`asap`) window. Use `standard` — see [Completion windows](/completion-windows).
</Warning>

Keep these rules in mind when routing through a LoRA:

- `model` must be listed in the LoRA's `supported_models`. Any other base model returns a `400`.
- A failed model validation blocks that model; `GET /v1/loras/{name}` exposes `validations[].result_message`.
- The LoRA must be owned by your organization. Names are per-org, so two organizations can each own a LoRA named `punchier-v1`.
- `metadata.lora` accepts either the LoRA's name or its UUID.
