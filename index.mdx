Most inference APIs are designed around a chat box: one short prompt, one fast answer. Agents don't work that way. They run for minutes or hours, fan out across many concurrent requests, and care far more about sustained throughput than about shaving milliseconds off any single call.

ValarAI is built for that second shape of workload. It serves open models across large fleets and schedules your traffic for aggregate throughput, so a job made of thousands of requests finishes sooner and costs less, even when an individual response takes a little longer to start.

## How requests work

ValarAI speaks the [OpenAI Responses API](https://developers.openai.com/api/reference/resources/responses/methods/create) at `/v1/responses`, reachable under the base URL `https://api.valarhq.ai/v1`. Any OpenAI-compatible client works once you repoint its base URL and key.

Because agent work is rarely interactive, the API leans on a `background` mode. Send `background: true` and the request returns a response id straight away instead of blocking; you then retrieve that id until the job reports `completed`. Pair that with [completion windows](/completion-windows) to trade latency for price, and you can keep large batches in flight without holding open a connection per request.

<Note>
  Authenticate with `Authorization: Bearer <key>`. Generate keys from the dashboard at [app.valarhq.ai](https://app.valarhq.ai), and store them in the `VALAR_API_KEY` environment variable so the SDK and OpenAI clients pick them up automatically.
</Note>
